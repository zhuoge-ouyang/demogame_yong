1 文件总览 —— 这个文件是干什么的？

在深入了解代码之前，先用大白话搞清楚这个文件的角色。

📄 文件身份

phase1-prompts.ts 是第一阶段所有 AI 生成任务的"指令库"，也可以叫它"出题册"。

它决定了"问 AI 什么问题"以及"要求 AI 怎么回答"。系统里所有第一阶段的世界观设定，都是由这份文件中的指令驱动 AI 来生成的。

🍳 生活化比喻：一套「菜谱目录」

想象一家西餐厅的后厨：

• phase1-prompts.ts 就是贴在墙上的菜谱目录——上面列出了10道菜的做法卡片

• 每张卡片（模块）上写着：菜名（moduleLabel）、做菜步骤（userPromptTemplate）、装盘要求（outputFormatInstructions）、先备好的食材（contextDependencies）

• 有的卡片还附带参考食谱（referenceDocuments），相当于大厨之前写过的草稿

• AI 就是那个按菜谱做菜的厨师——你给它什么菜谱，它就做什么菜

🔗 在系统中的角色

这个文件是整个 AI 辅助创作系统的起点。当用户在前端界面点击"AI生成"时：

1. 系统根据当前模块找到对应的菜谱卡片

2. 把卡片上的所有信息组装成一条完整的指令（Prompt）

3. 发给 AI，AI 按指令输出结果

4. 结果填回编辑界面，供人审阅修改

简而言之：没有这份文件，AI 就不知道要写什么、怎么写。

3 PromptTemplate 数据结构 —— 每道题长什么样

每个模块都遵循统一的"菜谱卡片"格式，下面逐个字段解释。

export interface PromptTemplate { templateId: string moduleLabel: string contextDependencies: string[] userPromptTemplate: string outputFormatInstructions: string referenceDocuments?: { label: string; content: string }[] }
字段名类型比喻含义白话说明
templateId string 题目编号 模块唯一标识 每道菜的唯一编号，比如"R-01"。系统通过这个编号找到对应的菜谱
moduleLabel string 题目标题 中文显示名 人看的菜名，比如"三界概述"，显示在界面上的标题
contextDependencies string[] 前置题目 依赖的前置模块列表 "做这题之前，先把这几题的答案准备好"。比如凡界设计依赖三界概述的答案
userPromptTemplate string 题目正文 发给 AI 的指令内容 完整的设计要求——告诉AI"你要设计什么、注意什么、包含哪些要素"
outputFormatInstructions string 答题模板 要求 AI 的回答格式 告诉AI"请用【标记】开头分段回答"——相当于答题卡上的填空格式
referenceDocuments 可选数组 开卷参考资料 附带的参考素材 有的模块附带了之前写过的草稿或素材，相当于"开卷考试，你可以参考这些资料"
📝 考试比喻

把每个 PromptTemplate 想象成一张考试卷：

• templateId = 准考证号（系统识别用）

• moduleLabel = 科目名称（语文、数学……）

• contextDependencies = "先修科目"（考高数前得先学过线代）

• userPromptTemplate = 大题的题目描述

• outputFormatInstructions = 答题格式要求（"请用总分总结构回答"）

• referenceDocuments = 允许带进考场的参考资料

🧱 比喻：盖楼

依赖关系就像盖楼的施工顺序：

• 三界概述是地基——没有它，上界/凡界/深渊都无从谈起

• 上界是一楼——建在地基上

• 凡界是二楼——地基+一楼都有了，才能建二楼

• 深渊是三楼——前两层都好了才盖

• 玩家身份→英雄系统→城堡女神是室内装修链——先定谁是业主（玩家），再设计家具（英雄），最后布置装饰（城堡女神）

⚠️ 关于 realm-structure 的特殊情况

realm-structure 虽然标记为 @deprecated（已废弃），但仍然是 main-storyline、player-identity、hero-system、castle-goddess、world-tree-system 五个模块的依赖源。

这是因为这些模块是在拆分之前创建的，它们依赖的是旧版大题的答案数据。拆分后 realm-structure 仍然保留在系统中，用于兼容已有数据——就像老菜单虽然不再推荐，但后厨仍然能照着做。

6 每个模块的详细解读

逐一拆解10个模块：做什么、依赖谁、参考什么、输出什么。

realm-structure 三界结构优化 已废弃

📌 模块作用

这是最初版本的"超级大题"，一次性设计上界、凡界、深渊三界的完整世界观。后来发现题目太大，AI 难以一次性高质量完成，所以拆分成了4个子模块（overview/upper/mortal/abyss）。

🔗 依赖（contextDependencies）

无依赖 [] —— 这是"地基题"，不需要先做任何其他题。

📎 参考文档（referenceDocuments）

无

📋 题目核心要点

  • 设计三界（上界/凡界/深渊）的过去、现状、未来
  • 上界必须有多派系分化（至少三个派系，核心要求）
  • 凡界九大元素大陆呈圆环布局，圣心城在中心
  • 深渊不是纯邪恶，而是世界树根系
  • 解释封魂令松动的原因
  • 三界由第一次大战形成

🏷️ 输出标记列表

【三界概述】 【玩家与英雄的身份】 【上界-过去】 【上界-现状】 【上界-未来】 【凡界-过去】 【凡界-现状】 【凡界-未来】 【深渊-过去】 【深渊-现状】 【深渊-未来】
🗣️ 白话总结

让 AI 一次性设计三界的完整世界观——从"怎么形成的"到"现在什么样"到"将来会怎样"。因为题目太大，后来拆成了4道小题。

realm-overview 三界概述 活跃

📌 模块作用

设计三界的整体格局概述——只写宏观框架，不深入每一界的细节。是从旧版 realm-structure 拆出来的"总论"部分。

🔗 依赖（contextDependencies）

无依赖 [] —— 拆分后仍然是最基础的"地基模块"，不依赖任何其他模块。

📎 参考文档（referenceDocuments）

无

📋 题目核心要点

  • 魂域如何分成三界（第一次大战的结果）
  • 9枚封魂令的来源、作用、为何松动
  • 魂能的解释及其在三界中的作用
  • 世界树尤克特拉希尔在三界中的不同表现
  • 三界之间的关联与对立关系
  • 玩家与英雄的身份概述

🏷️ 输出标记列表

【三界概述】 【玩家与英雄的身份】
🗣️ 白话总结

让 AI 写一篇"三界世界观的概述导言"——宏观地介绍世界是怎么来的、封魂令是什么、魂能是什么、三界什么关系，以及玩家和英雄大概是什么角色。相当于一本书的"引言"章节。

realm-upper 上界·神殿 活跃 含参考文档

📌 模块作用

在上界概述的基础上，深入设计上界/神殿的详细世界观——派系、神祗、过去/现状/未来。

🔗 依赖（contextDependencies）

['realm-overview'] —— 必须先完成三界概述，才能设计上界细节。因为上界是三界之一，必须和总论保持一致。

📎 参考文档（referenceDocuments）

本模块是唯一附带了参考文档的模块，包含3份素材：

  • 上界·神殿——过去（参考稿）：魂域大战前的众神、大战中的众神、三界裂变后的上界
  • 上界·神殿——现状（参考稿）：世界树在上界的表现、神殿四大派系（秩序圣庭/深渊执剑团/源流议会/星幕隐庭）的详细设定
  • 上界·神殿——未来（参考稿）：上界的命运分岔——最优/次优/灾难三种路径

💡 相当于"开卷考试"——AI 可以参考之前写好的上界素材来优化输出。

📋 题目核心要点

  • 神界派系分化（至少3个派系，核心要求）
  • 重要神祗的个人情感、力量定位、经历
  • 封魂令松动与上界格局变化的关系
  • 参考已提供的素材进行优化和完善

🏷️ 输出标记列表

【上界-过去】 【上界-现状】 【上界-未来】
🗣️ 白话总结

让 AI 基于三界概述，深入设计上界神殿的世界观——有哪些派系、各派系怎么斗、关键神是谁，并参考之前写好的草稿来优化。重点要求：神殿必须有派系博弈，不是一团和气。

realm-mortal 凡界 活跃

📌 模块作用

在三界概述和上界设定的基础上，设计凡界的详细世界观——这是玩家冒险的核心舞台。

🔗 依赖（contextDependencies）

['realm-overview', 'realm-upper'] —— 需要三界概述的宏观框架，也需要上界设定（因为上界的封魂令松动直接影响凡界）。

📎 参考文档（referenceDocuments）

无

📋 题目核心要点

  • 九大元素大陆：金、木、冰、火、土、风、雷、光、暗
  • 圣心城：九大陆环心的核心枢纽城市
  • 符文通道：连接各大陆的古老传送网络
  • 每个大陆有9个区域
  • 封魂令松动与凡界格局变化的关系

🏷️ 输出标记列表

【凡界-过去】 【凡界-现状】 【凡界-未来】
🗣️ 白话总结

让 AI 设计玩家冒险的主舞台——凡界。重点是九大陆的布局和特色、圣心城的中枢地位、封魂令松动引发的危机。需要和上界设定保持一致。

realm-abyss 深渊 活跃

📌 模块作用

在三界概述、上界和凡界设定的基础上，设计深渊的详细世界观——三界链的最后一环。

🔗 依赖（contextDependencies）

['realm-overview', 'realm-upper', 'realm-mortal'] —— 深渊是依赖最多的模块，需要三界概述的框架、上界的封魂令背景、凡界的现状，才能设计与之呼应的深渊。

📎 参考文档（referenceDocuments）

无

📋 题目核心要点

  • 深渊的本质：不是纯邪恶，而是世界树的根系网络
  • 原始魔法的源泉，代表循环与平衡
  • 深渊有自己的生态和文明
  • 封魂令松动与深渊格局变化的关系
  • 与上界、凡界形成三界互为关联的完整体系

🏷️ 输出标记列表

【深渊-过去】 【深渊-现状】 【深渊-未来】
🗣️ 白话总结

让 AI 设计三界中最神秘的深渊——它不是反派大本营，而是世界树的根系，有自己的文明。设计时必须确保深渊和上界、凡界形成闭环，不能割裂。

main-storyline 凡界主线剧情 活跃

📌 模块作用

基于三界世界观，设计凡界范围内的主线剧情三章节（过去/现状/未来），每章包含核心事件、关键角色、玩家参与方式。

🔗 依赖（contextDependencies）

['realm-structure'] —— 剧情必须基于世界观来写，而此模块创建时用的还是旧版 realm-structure（尚未迁移到拆分后的子模块）。

📎 参考文档（referenceDocuments）

无

📋 题目核心要点

  • 三章节结构：过去（繁荣但暗流）→现状（封印松动）→未来（危机爆发与抉择）
