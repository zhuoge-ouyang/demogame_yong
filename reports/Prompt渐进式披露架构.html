两套上下文机制并存

系统同时运行两套上下文收集机制，根据模块的 contextDependencies 配置自动选择：

子模块级（新）推荐

  • 适用于 三界子模块（realm-overview / upper / mortal / abyss）
  • 从 Store 直接提取依赖子模块的完整字段内容
  • 区分定稿 / 草稿 / 参考文档三种级别
  • 函数：collectModuleContext()

类别级（原有兼容）兼容

  • 适用于 其他模块（剧情 / 人设 / 世界树）
  • 从 moduleSummaries 提取分类摘要
  • 按类别（setting / storyline / character / worldview）聚合
  • 函数：collectSummaryContext()
自动选择逻辑：系统检测 contextDependencies 中是否包含子模块 ID（在 SUB_MODULE_FIELD_MAP 中注册）。若包含，使用子模块级收集；否则使用类别级收集。

四大内容类别

所有模块按内容性质归入四大类别，每个类别有独立的系统提示词层和上下文需求配置：

类别 ID 中文名 包含模块 系统提示词层 上下文需求
setting 设定模块 realm-overview, realm-upper, realm-mortal, realm-abyss base + setting 无（基础源头）
worldview 世界观模块 world-tree-system base + setting setting critical
storyline 剧情模块 main-storyline base + narrative setting critical + worldview high
character 人设模块 player-identity, hero-system, castle-goddess base + narrative setting critical + storyline high

2渐进式定稿工作流程

用户按顺序依次生成各模块内容，每一步都基于前序已定稿的内容，确保世界观层层递进、互不矛盾。

1

生成「三界概述」realm-overview

首先生成三界的整体格局概述，包含三界来历、封魂令、魂能、世界树、三界关系、玩家与英雄身份概要。这是所有后续模块的基础。

注入内容：无外部上下文依赖
  • 仅使用系统提示词（base + setting 层）+ 用户模板 + 输出格式

→ 审阅编辑 → 定稿 ✓

2

生成「上界·神殿」realm-upper

基于定稿的三界概述，详细设计上界神殿的过去、现状与未来。重点关注神界派系分化和重要神祗设定。

自动注入：
  • 已确认设定 定稿的三界概述（realm-overview 全文）
  • 参考素材 上界·神殿三篇参考稿（过去/现状/未来）

→ 审阅编辑 → 定稿 ✓

3

生成「凡界」realm-mortal

基于定稿的概述和上界设定，详细设计凡界的过去、现状与未来。重点关注九大元素大陆、圣心城和封魂令松动的影响。

自动注入：
  • 已确认设定 定稿的三界概述（realm-overview 全文）
  • 已确认设定 定稿的上界全文（realm-upper: 过去/现状/未来）

→ 审阅编辑 → 定稿 ✓

4

生成「深渊」realm-abyss

基于定稿的概述、上界和凡界设定，详细设计深渊的过去、现状与未来。确保深渊设计与三界体系形成完整闭环。

自动注入：
  • 已确认设定 定稿的三界概述（realm-overview 全文）
  • 已确认设定 定稿的上界全文（realm-upper: 过去/现状/未来）
  • 已确认设定 定稿的凡界全文（realm-mortal: 过去/现状/未来）

→ 审阅编辑 → 定稿 ✓

Prompt 组装公式

每一步生成时，系统按以下顺序组装完整的用户提示词：

系统提示词
+
已确认设定
（严格遵循）
+
参考内容
（可优化）
+
参考素材
（可借鉴）
+
用户模板
+
输出格式
系统提示词 — 根据模块类别分层注入
已确认设定 — 定稿内容，标记为"严格遵循，不可修改"
参考内容 — 草稿内容，标记为"可在此基础上优化"
参考素材 — .md 参考文档，标记为"可借鉴风格和设定方向"
用户模板 — 当前模块的任务描述
输出格式 — 【标记】格式化输出要求
核心原则：只有"已定稿"的内容才会以"严格遵循"级别注入后续模块。草稿内容以"参考"级别注入，AI 可以在此基础上优化。这种区分保证了世界观一致性的同时，保留了灵活性。

4上下文注入机制说明

深入理解两种上下文收集机制的工作原理，以及三级内容区分如何影响 AI 的生成行为。

子模块级收集新机制

函数：collectModuleContext()

工作原理：

  • 遍历 template.contextDependencies 中的子模块 ID
  • 通过 SUB_MODULE_FIELD_MAP 查找对应的 store 数据键和字段列表
  • 从 worldStore.state 直接提取依赖子模块的完整字段内容
  • 逐字段检查定稿状态（isFieldFinalized()）
  • 若该子模块所有字段均已定稿 → 归入 finalizedParts
  • 若存在未定稿字段 → 归入 draftParts
  • 额外注入 template.referenceDocuments 中的参考文档
// SUB_MODULE_FIELD_MAP 映射表 'realm-overview': { storeKey: 'realmStructure', fields: ['summary'], label: '三界概述' } 'realm-upper': { storeKey: 'realmStructure', fields: ['upper.past', 'upper.present', 'upper.future'], label: '上界·神殿' } 'realm-mortal': { storeKey: 'realmStructure', fields: ['mortal.past', 'mortal.present', 'mortal.future'], label: '凡界' } 'realm-abyss': { storeKey: 'realmStructure', fields: ['abyss.past', 'abyss.present', 'abyss.future'], label: '深渊' }

类别级收集兼容

函数：collectSummaryContext()

工作原理：

  • 根据当前模块类别，获取 getCategoryContextConfig() 配置
  • 遍历 contextNeeds 列表中的需求项
  • 对每个需求项，调用 extractModuleSummary() 从原始数据提取精简摘要
  • 摘要截断到 maxSummaryChars 字符
  • 通过 checkCategoryFinalized() 检查来源类别的定稿状态
  • 归入 finalizedParts 或 draftParts
  • 无参考文档注入能力
// 类别上下文需求配置 setting: contextNeeds: [] // 基础源头，无需上下文 worldview: contextNeeds: [{ sourceCategory: 'setting', maxSummaryChars: 500, priority: 'critical' }] storyline: contextNeeds: [{ sourceCategory: 'setting', maxSummaryChars: 500, priority: 'critical' }, { sourceCategory: 'worldview', maxSummaryChars: 300, priority: 'high' }] character: contextNeeds: [{ sourceCategory: 'setting', maxSummaryChars: 500, priority: 'critical' }, { sourceCategory: 'storyline', maxSummaryChars: 300, priority: 'high' }]

三级内容区分

上下文注入后，在用户提示词中按三个层级标注，让 AI 明确不同内容的处理方式：

🔒

已确认设定

来源：已定稿的模块内容

"请严格遵循，不可修改"

AI 生成的新内容必须与这些设定完全一致，不得产生矛盾

📝

参考内容

来源：草稿（未定稿）的模块内容

"可在此基础上优化"

AI 可以参考这些内容的方向，但允许调整和优化

📎

参考素材

来源：.md 参考文档

"可借鉴风格和设定方向"

仅作为风格参考，当与已确认设定冲突时以已确认设定为准

// 实际 Prompt 中的标注格式 --- 以下为已确认的设定（请严格遵循，不可修改）--- === 三界概述（已有内容） === （定稿的概述内容...） --- 以下为参考内容（可在此基础上优化）--- === 上界·神殿（已有内容） === （草稿内容...） --- 以下为参考素材（可借鉴风格和设定方向，但以已确认设定为准）--- === 上界·神殿——过去（参考稿） === （参考文档内容...） 【重要约束】对于已确认的设定，请严格保持一致，不要修改或矛盾。

5参考文档机制

参考文档是预先准备的优质设定素材，作为"风格参照"注入特定子模块的 Prompt，帮助 AI 生成更高质量的内容。

references/ 目录结构

src/services/prompts/references/
├── upper-realm-refs.ts  ← 上界·神殿参考文档（当前已有）
├── mortal-realm-refs.ts  ← 凡界参考文档（待扩展）
└── abyss-realm-refs.ts  ← 深渊参考文档（待扩展）

当前已有：上界·神殿三篇参考稿

存储在 upper-realm-refs.ts 中，导出为 UPPER_REALM_REFERENCES 对象，包含三个属性：

属性 标签名 注入到 内容概要
UPPER_REALM_REFERENCES.past 上界·神殿——过去（参考稿） realm-upper 魂域大战前的众神、三界裂变后的上界
UPPER_REALM_REFERENCES.present 上界·神殿——现状（参考稿） realm-upper 世界树在上界的表现、神殿四大派系
UPPER_REALM_REFERENCES.future 上界·神殿——未来（参考稿） realm-upper 上界未来展望

如何扩展：新增凡界/深渊参考文档

扩展参考文档只需 3 步：

// 第一步：在 references/ 目录下创建新文件 // 例如：mortal-realm-refs.ts export const MORTAL_REALM_REFERENCES = { past: `（凡界过去参考稿内容）`, present: `（凡界现状参考稿内容）`, future: `（凡界未来参考稿内容）` } // 第二步：在 phase1-prompts.ts 中引入 import { MORTAL_REALM_REFERENCES } from './references/mortal-realm-refs' // 第三步：在 realm-mortal 模板配置中添加 referenceDocuments 'realm-mortal': { ... referenceDocuments: [ { label: '凡界——过去（参考稿）', content: MORTAL_REALM_REFERENCES.past }, { label: '凡界——现状（参考稿）', content: MORTAL_REALM_REFERENCES.present }, { label: '凡界——未来（参考稿）', content: MORTAL_REALM_REFERENCES.future } ] }
参考文档来源：通常由甲方/策划团队提供 .md 或 .txt 格式的参考稿件，开发人员将其转为 TypeScript 常量导出。参考文档的优先级为 medium，在 Token 紧张时会优先裁剪。

6Token 管理

渐进式注入意味着上下文会随模块推进而累积。Token 管理策略确保 Prompt 始终在模型的上下文窗口限制内。

渐进式注入带来的 Token 累积问题

以四步生成流程为例，上下文的 Token 消耗逐步增长：

生成步骤 注入的上下文 上下文量级趋势
第1步 三界概述 无外部上下文 最少 — 仅有系统提示词和模板
第2步 上界·神殿 概述（定稿）+ 上界参考稿（3篇） 增加 — 概述 + 参考文档
第3步 凡界 概述（定稿）+ 上界（定稿） 继续增长 — 累积定稿内容
第4步 深渊 概述（定稿）+ 上界（定稿）+ 凡界（定稿） 最大 — 三个模块的定稿内容叠加
注意：第2步（上界）因包含参考文档，上下文量可能超过第3步。但参考文档优先级为 medium，在 Token 紧张时会被裁剪。

Token 预算分配

系统将模型的上下文窗口按 60% / 20% / 20% 分配给三个部分：

上下文 60%
模板 20%
输出 20%
上下文（60%）— 已确认设定 + 参考内容 + 参考素材
模板（20%）— 系统提示词 + 用户模板 + 输出格式
输出（20%）— 预留给 AI 生成内容的窗口

结构化裁剪策略

当上下文 Token 数超过预算时，系统按优先级从低到高逐级裁剪，每个部分按 Markdown 段落裁剪而非粗暴字符截断：

low 完全移除 — 低优先级内容直接替换为"（已省略低优先级上下文）"
medium 裁剪至 200 字符 — 按 Markdown 段落保留首段，参考素材属于此级别 ⚠️
high 裁剪至 400 字符 — 按 Markdown 段落保留标题和首段
critical 不裁剪 — 核心设定内容始终保留，确保世界观一致性 ✅
参考素材优先级为 medium：在 Token 紧张时，参考文档会优先于定稿内容和草稿内容被裁剪。这意味着 AI 仍会严格遵循已确认设定，但可能丢失参考文档中的风格引导。

Markdown 段落裁剪（而非字符截断）

传统的字符截断会从中间切断段落，导致语义不完整。系统使用 trimByParagraphs() 按段落边界裁剪：

// 裁剪策略 1. 按 \\n\\n 或 \\n# 分割段落 2. 保留第一个段落（通常是标题/概述） 3. 逐步添加后续段落，直到超出字符限制 4. 非标题段落：可部分添加（剩余空间 > 50 字符时） 5. 标题段落：不部分添加（避免不完整标题） 6. 末尾添加省略提示："（已省略 N 个段落的详细内容）"
优势：段落级裁剪保证了每个保留的段落语义完整，AI 能正确理解上下文意图，不会因截断产生歧义。

Token 估算方法

系统使用 estimateTokenCount() 进行快速 Token 估算，无需实际调用分词器：

// 估算公式 中文字符数 × 2.5 + 英文单词数 × 1.5 + 其他字符数 × 0.5 // 示例：一段 500 中文字的内容 500 × 2.5 = 1,250 tokens（估算值）
注意：此为估算值，实际 Token 数取决于各模型的具体分词器。对于中文内容，估算通常偏高 5-10%，但足以用于预算控制。
