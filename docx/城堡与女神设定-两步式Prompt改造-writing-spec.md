## 1. 背景与目标

### 1.1 问题陈述
当前「城堡与女神设定」模块采用**单次 Prompt 一次性生成 10 个字段**的模式：

- 单次 Prompt 过长，AI 难以兼顾所有字段
- 上游已定稿的【神界现状】【玩家身份】【凡界现状】设定无法被充分锚定
- 生成结果频繁出现设定漂移：
  - 城堡被写成"人类建造的要塞"（应为世界树意志生成的活体建筑/玩家身体）
  - 方舟堡位置被写成"圣心城废墟"（应为翠森大陆深处的根核节点）
  - 女神被写成"普通新手引导 NPC"（应为源流议会领袖+三界存续守护者）
  - 女神外观缺失核心视觉符号（光翼/翅膀）
  - 女神被描写为"会消失/昏迷/偶尔不在场"（违反"神祇引导者恒定共存"原则）
  - 出现"第十魂令"等凡界前期剧情禁用概念

### 1.2 改造目标
- **保证一致性**：显式锚定上游已定稿的 5 个关键模块设定
- **保证完整性**：10 字段输出质量均衡，不出现某字段详实、另字段敷衍
- **保证恒定性**：女神形象严格符合"恒定共存 + 能力降级"原则
- **保证可迭代**：用户可在纲要阶段审阅/微调，避免一整次完整生成被全部推翻

### 1.3 技术方案总览
采用**真两次 AI 调用的两步式生成流程**（参考已有 `main-storyline-overview` + `main-storyline-chapter1/2/3` 先例）：

```
Step 1: castle-goddess-outline   → 生成设定纲要（非正式字段，不写入 world.json）
Step 2: castle-goddess-detail    → 基于纲要扩展为 10 个正式字段（写入 world.json）
```

| # | 约束 |
|---|---|
| R1 | 城堡 ↔ 玩家：载体-意识（身体与灵魂），不是"居住者-居所" |
| R2 | 女神 ↔ 方舟堡：共存关系，女神是方舟堡的"守望神像" |
| R3 | 女神 ↔ 玩家：长伴引导者与被引导的种子，非时隐时现 |

- "圣心城"、"圣心城废墟"、"祈光塔"作为方舟堡位置
- 把方舟堡写成"玩家居住的基地"、"人类建造的要塞"

- 描写女神"消失/昏迷/离开/被迫退场/偶尔不在场"
- 女神外观缺失"光翼/翅膀"核心视觉符号

**配置项**
```yaml
moduleId: castle-goddess-outline
moduleLabel: 城堡与女神-设定纲要
stepOrder: 1
contextDependencies:
  - realm-overview
  - realm-upper
  - realm-mortal
  - player-identity
  - hero-system

outputTargetFields: []   # 不写入 world.json 任何字段
```

**userPromptTemplate**（完整版见附录 A.1）

**outputFormatInstructions**
```
请按以下格式输出纲要，条目短句，不要展开：

【城堡锚点】
- ...（3-5 条）

【女神锚点】
- ...（5-8 条）

【关系锚点】
- ...（3 条）

严禁输出正式字段（如 description/role/name 等），只输出锚点短句。

  - realm-mortal
  - player-identity
  - hero-system
contextMaxChars: 1500
extraContext:
  - outline   # Step1 纲要，通过 {{outline}} 占位符注入
outputTargetFields:
  - castleGoddess.castle.description
  - castleGoddess.castle.role
  - castleGoddess.castle.significance
  - castleGoddess.goddess.name
  - castleGoddess.goddess.appearance

**userPromptTemplate**（完整版见附录 A.2）

**outputFormatInstructions**（保留现有【标记】格式，兼容现有 parser）
```
请严格按以下格式输出，每段内容用【标记】开头，段与段之间空一行：

【城堡描述】...
【城堡作用】...
【城堡意义】...
【女神名字】艾蕾尼娅（Elennia）
【女神外观】...
【女神性格】...
【女神身份】...
【女神引导】...

## 4. 前端改造设计

### 4.1 组件结构

```
CastleGoddess.vue
├── [新增] Step 切换器（Segmented Tab）
│   ├── Step 1: 设定纲要
│   └── Step 2: 完整详述（纲要未确认时 disabled）
├── [新增] 纲要预览/编辑卡片（Step 1 完成后显示）
│   ├── 纲要文本（可编辑 textarea）
│   ├── 状态：draft / confirmed
│   └── 操作：重新生成 / 确认作为 Step2 输入 / 清空
├── [保留] 现有 10 字段表单（CollabTextField × 10）
└── [保留] 现有定稿/解锁按钮逻辑
```

### 4.2 纲要存储策略

| 维度 | 方案 |
|---|---|
| 存储位置 | `localStorage['castleGoddessOutline']` + Vue 组件 ref |
| 存储结构 | `{ text: string; generatedAt: number; confirmed: boolean }` |
| 是否同步后端 | 否（过程数据，不纳入数据契约） |
| 是否纳入定稿机制 | 否（finalized 只管正式字段） |
| 是否导出 | 否（导出 world.json 时忽略） |
| 清理时机 | Step2 成功写入 10 字段后，提示用户可选清空 |

### 4.3 交互流程

```
用户进入"城堡与女神"页面
  ├─ 默认选中 Step 1
  ├─ 【Step 1 视图】
  │   ├─ 点击 AI 生成 → 调用 castle-goddess-outline
  │   ├─ 纲要返回 → 展示在可编辑卡片
  │   ├─ 用户微调（可选）
  │   └─ 点击"确认作为 Step2 输入" → confirmed=true
  ├─ 【Step 2 视图】（纲要未确认时 disabled + tooltip）
  │   ├─ 顶部显示当前纲要摘要（折叠面板）
  │   ├─ 点击 AI 生成 → 调用 castle-goddess-detail
  │   │   （自动注入 localStorage 中纲要到 {{outline}} 占位符）
  │   ├─ 返回 10 字段 → 填入表单
  │   └─ 用户审阅、微调、逐字段定稿
  └─ 所有字段定稿 → 与原流程一致
```

### 4.4 边界 Case

| # | 场景 | 处理 |
|---|---|---|
| E1 | 用户直接点 Step 2 而未生成纲要 | 按钮 disabled + tooltip 提示 |
| E2 | 纲要已编辑未点"确认"就切 Step 2 | 弹窗："当前纲要未确认，是否使用当前编辑内容？" |
| E3 | 纲要过长（>5000 字） | textarea maxlength + 提示 |
| E4 | Step 2 失败回退 | 已确认纲要不清除，用户可直接重试 |
| E5 | 跨 session 恢复 | localStorage 保存，页面重入自动恢复 |
| E6 | 用户完成 Step2 后想重新 Step1 | 提供"清空纲要并重新开始"按钮 + 确认弹窗 |

## 6. world.json 数据修订

### 6.1 `castleGoddess.castle.description`
**变更原因**：当前文本含"坐落于圣心城废墟的中心"，与主线剧情（方舟堡初始位置为翠森大陆深处）冲突。

**修订方向**：保留方舟堡=活体建筑、玩家身体、世界树根脉共生等核心表述；删除"圣心城废墟"等城市锚点；增加"翠森大陆深处的根核节点"初始位置。

（具体文本由 writing-plans 阶段产出。）

### 6.2 `castleGoddess.goddess.appearance`
**变更原因**：
1. 缺失参考图中的"光翼/翅膀"核心视觉符号
2. 现有"越是隐瞒，越是透明"可能被误读为"消失"（违反存在恒定原则）
3. 需要明确"常驻方舟堡最高塔楼顶端"位置锚点

**修订方向**（建议终稿，可在 writing-plans 阶段微调）：
> 她以半透明的光影形态常驻于方舟堡最高塔楼的顶端，如一尊永不离位的守护神像——舒展的光翼从她肩后绽开，金色羽翎与柔光交织，身姿挺立面朝远方。她的轮廓优雅而略显模糊，如同阳光穿过薄雾后留下的温暖残影；长发如流动的星光，面容在美丽与哀愁之间徘徊。当她神力衰弱时，光翼会收敛黯淡、身影变得更加沉静，但从不消失——她始终在城堡上方守望。

### 6.3 修订执行方式
- 由于涉及已 `finalized` 字段，需要实施阶段：
  1. 先解锁（或提供一次性管理员解锁脚本）
  2. 替换文本
  3. 重新定稿（保持 finalized 状态）
- 参考前端自动保存覆盖 world.json 的经验，修订优先在服务停止状态下直接编辑 JSON，避免前端覆盖。

## 硬约束（不可违反的上游已定稿锚点）

### 关于城堡（方舟堡）
1. 城堡的唯一名称为"方舟堡"，非人工建造，而是世界树意志生成的活体建筑
